Architecture Flow Diagram
The browser interface sends HTTP requests to the PHP backend and polls it every 5 seconds for status updates. The PHP backend is the control plane: it starts and stops the Python orchestrator, reads system metrics, checks the local and remote LLM APIs, and manages the PHP-side storage used for orchestrator profiles, launcher files, PID tracking, and metadata. The Python orchestrator is the execution engine: it runs the A/B loop, applies guardrails, queries the LLM nodes through llama-server endpoints, and writes run outputs into the runs storage path. The system block represents monitoring inputs used by the PHP backend, including /proc, /v1/models, and health checks. The frontend never talks directly to the LLM nodes.
Execution Notes
HTTP Sync Layer
Initial page load, profile read/save, start and stop actions are synchronous exchanges between browser and PHP.
Polling Layer
The frontend polls status.php and run-status.php every 5 seconds to refresh health data, run state, and transcript entries.
Process Execution Layer
run-start.php launches the Python orchestrator with nohup, then writes PID and launcher metadata used by subsequent status checks.
LLM Exchange Layer
The orchestrator alternates POST requests to each llama-server endpoint, applying prompt history rules and output constraints for every turn.
Persistence Layer
PHP storage keeps profiles and launcher metadata, while the Python orchestrator writes run folders and transcripts into the runs output path for traceability and replay.
